## 0. Why this is a v2 (and not a v1.1)

The first program docs were intellectually coherent but architecturally over-built. An external architectural review (`docs/superpowers/2026-05-29-architect-review-analysis.md`) and four parallel OSS-landscape investigations surfaced the same conclusion from different angles: the previous plan re-implemented infrastructure that already exists in well-maintained open source, while under-investing in the one thing nobody else has built — rule-effectiveness measurement for AI coding tools.

This v2 keeps every piece of intellectual capital that earned its place (the detector scenarios, the Phase 0.5 prototype, the use cases, the OSS comparison) and throws away the speculative scaffolding (custom FastAPI + SQLite + ChromaDB tier, Apps Script mailer, bespoke release-notes generator, 14-detector taxonomy as a single monolith).

The result is a smaller program, a sharper contribution, and a much shorter document.

### 3.3 Issue-to-trace back-reference (the triage-pivot contract)

When the central team triages an Issue, they need to pivot from "developer says X happened" to "here is the actual session that prompted this." That link is a structured field in the Issue body, populated by both paths and read by the triage UI:

- **Manual feedback (Phase 1):** the `/ai-sherpa-feedback` skill captures the current session ID (from `$env:CLAUDE_SESSION_ID` or the SessionEnd hook context) and writes it into `session_ref` per `feedback_issue.schema.json`.
- **Auto-filed feedback (Gate E):** the Scorer Registry's webhook handler writes the Langfuse trace URL + sample-session IDs into the same `session_ref` field.
- **Triage UI:** the Phase 1 Issue template renders `session_ref` as a clickable Langfuse deep-link (or, for manual-only Phase 1, as a plain session-ID hint for later correlation). If absent — because the developer redacted it — triage falls back to keyword search and the Issue is implicitly less actionable.

Without `session_ref`, every manual triage decision is "we take their word for it" and every auto-filed finding is a claim with no exhibit. This is the one cross-phase contract that has to be right on day 1 of Phase 1, because Gate E inherits it.

## 4. The program shape — gates, not phases

A phase is a time-boxed deliverable. A gate is a conditional transition. The previous program was a sequence of phases with predetermined deliverables; this one is a sequence of gates with explicit go/no-go criteria. The difference matters when reality intrudes.

### Gate A — Spike (week 0–1, 5 working days)

**Run in parallel:**
- Spike 1: Anthropic native OTel export — content coverage analysis (3 days)
- Spike 2: Anthropic Admin API — schema, auth, rate limits (3 days)
- Spike 3: Infra capability — Docker on existing Windows server vs Linux VM provisioning (1 day, ops question)

**Exit criteria:** a one-page decision matrix documenting what each external dependency actually delivers vs what was assumed.

**Kill criterion:** if both OTel coverage and Admin API turn out empty, the Scorer Registry approach needs re-scoping before Phase 1 — escalate before Gate B.

### Gate B — Phase 1 (weeks 2–4, ~3 weeks)

**Ships:** manual feedback intake + GitHub-native release pipeline. No custom mailer. No bespoke release-notes generator. Adopts GitHub native tooling end-to-end. (Detailed deliverables in §5.)

**Exit criteria:**
- All Phase 1 artifacts merged to master and used in a real `v2026.06.dd` release
- `schemas/` directory checked in and validated in CI
- Rollback runbook executed in a dry run on a fork
- On-call rota documented and assigned

**Kill criterion:** if branch-protection / push-back issues prevent the release Action from working on a public org repo (per the architect review's risk #4), redesign the release mechanism — do not paper over with `--no-verify`.

### Gate C — Phase 1 stabilization (weeks 5–8, ~4 weeks)

Phase 2 is **not scoped during this window.** The goal is to learn whether central-team triage scales.

**Exit criteria (all must hold for 4 consecutive weeks):**
- Triage backlog age < 5 days on manual feedback
- Release Action success rate ≥ 95%
- ≥ 20 Issues triaged per week (proves intake works)
- Zero P0 incidents from release rollouts

**Kill criterion:** if any of these fails for 2 consecutive weeks, Phase 2 is deferred until they hold. Building a telemetry firehose into a clogged manual sink is the failure mode the previous plan would have produced.

### Gate D — Phase 2 scoping (week 9, 5 working days)

Revisit the Phase 2 scope against actual Gate A + Gate C data. The detector list (currently 14 in the archived plan) is re-derived from what Phase 1 triage *actually* surfaced as common failures. We scope what we know we need, not what we speculated about a quarter ago.

**Exit criteria:** a Phase 2 spec ≤ 400 lines with explicit kill criteria of its own.

### Gate E — Phase 2 build (weeks 10–13, ~4 weeks)

**Ships:**
- Adopted stack deployed (OTel Collector laptop + gateway, Langfuse, ClickHouse) — config, not custom code
- Scorer Registry with N rules drawn from Phase 1 evidence (N likely 4–6, not 14)
- Webhook handler that auto-files into Phase 1's triage queue
- Privacy controls: NDA OTTL filter, per-machine opt-out flag, PII scrubber with test corpus

**Kill criterion:** if Scorer Registry false-positive rate > 30% in the first 2 weeks of production data, auto-filing is gated behind manual approval and the program pauses for Scorer tuning.

### What there is no Phase 2b for

Langfuse ships built-in LLM-as-judge eval primitives. "LLM polish of findings" is one Scorer in the registry, not a separate phase. If the cost ceiling becomes meaningful (>$5/wk), the existing Langfuse evals can be cost-capped via standard config; we don't build a bespoke `cost_report.py` CLI.

### 5.7 Testing strategy for the Scorer Registry (decided now, executed at Gate E)

Every Scorer in `scorers/rule_*.py` is a software artifact with the same discipline as production code. The previous program treated detectors as scripts; v2 treats them as components.

Per-Scorer requirements:

- **Golden corpus** (`scorers/<rule>/fixtures/`): 20–50 hand-labeled session excerpts per rule — synthetic or hand-sanitized, **never raw developer data**. Labels: `should_fire`, `should_not_fire`, `boundary`.
- **Property tests** (run on every PR touching the Scorer): known true-positives must score above threshold; known true-negatives must score below.
- **Calibration job** (nightly): runs the Scorer over the prior 7 days of production traces; emits precision / recall / F1 per Scorer to a Langfuse dashboard owned by the central team.
- **Regression gate**: a PR landing a Scorer change must show no precision regression vs the prior week's baseline. CI runs corpus + calibration replay.
- **Shadow mode**: new Scorers run for 14 days scoring but not firing webhooks. Auto-file eligibility requires shadow-mode F1 ≥ 0.7 or an explicit central-team override (recorded in the Scorer's README).
- **Adversarial corpus** (`scorers/<rule>/adversarial.jsonl`): known-hard cases (rule mentions `malloc` but in a comment; rule should not fire in a different-domain context that happens to mention an ISR). Adversarial accuracy stays ≥ 80% or the Scorer is downgraded out of auto-filing.
- **Fixture privacy**: CI fails on any fixture containing a PII regex match (email, path, SHA, GitHub handle).

System-level requirements:

- **Calibration history is versioned**: every nightly calibration writes a row to `scorer_calibrations` keyed on `(scorer_id, scorer_version, calibration_date)` so regressions can be bisected.
- **Replay determinism**: the same trace + same Scorer version must produce the same score. LLM-as-judge Scorers cache verdicts keyed on `(trace_hash, prompt_hash, model_version)` to keep CI deterministic.

### 5.9 What happens to the Phase 0.5 prototype

The prototype at `analyzer-prototype/` shipped, ran against real data, and produced 12 findings. It is **not** dead code in v2 — its role changes.

**Decision: keep as a standalone tool; lift its modules into the Scorer Registry.**

- **Standalone tool**: rename `analyzer-prototype/` → `tools/local-session-analyzer/`. Package as a CLI any individual developer can run against their own `~/.claude/projects/`. No fleet telemetry. No opt-in dance. This is a DX affordance (§11), not a successor product.
- **Lifted modules**: the metric functions (Read:Edit ratio, self-admitted error rate, repeat-edit detection, etc.) move into `scorers/generic_metrics.py`. The standalone tool and the fleet Scorer Registry share the same scoring logic, parameterized on corpus size.
- **Retired**: the prototype's HTML renderer + Jinja2 templates + sortable JS table. Standalone gets `df.to_markdown()`; fleet uses Langfuse's UI.
- **Versioning**: the standalone tool tracks its own version independently of the fleet Scorer Registry — a dev with last quarter's standalone tool keeps getting personal insights even if the fleet platform changes shape.

This decision honors the prototype's ship status without letting it grow into a product it was never scoped to be.

### 5.10 Archive mechanics for the v1 docs

The superseded specs and plans in §10 must be archived in a way that preserves history, prevents accidental re-use, and doesn't rot inbound links silently.

**Procedure (single commit):**

1. `git mv` (not `cp`) each archived file from `docs/superpowers/specs/` and `docs/superpowers/plans/` to `docs/superpowers/archive/2026-05-30-pre-v2/`. `git mv` preserves rename history; `cp` orphans it.

   ```
   > **ARCHIVED 2026-05-30.** Superseded by [docs/superpowers/2026-05-30-program-v2.md](../../2026-05-30-program-v2.md).
   > Preserved for historical context. Not authoritative. Do not cite.
   ```
3. Add `docs/superpowers/archive/README.md` with the archive policy (one paragraph: "docs here are reference-only; the live program brief lives at `../2026-05-30-program-v2.md`").
4. Grep the repo for inbound references (`grep -r "2026-05-29-phase2a" --include="*.md" .` etc.) and rewrite the still-load-bearing ones; leave the others as link-rot — that's fine for archived material.

**What NOT to do:**
- Don't `rm` the old files. They captured real thinking; the trail matters for postmortems and for new joiners.

## 8. Use cases — reduced and tightened

The previous requirements summary had 7 use cases, several of which oversold what the specs delivered. v2 keeps 4 core cases that match what the specs actually build, and adds 4 operational cases the previous summary missed.

### Core flows
- **UC1** Developer reports a Claude mistake via `/ai-sherpa-feedback` — same as before.
- **UC2** System detects a pattern via the Scorer Registry and files a candidate-change Issue — *language change*: the candidate is a *suggested* rule edit, not an *executed* one; central team review is required.
- **UC3** Central team triages and ships a weekly release — same flow, simpler tooling.
- **UC4** Developer gets the update via Atom-feed email + `setup --update` — adds explicit `--pin` flag.

### Operational flows (missing before — add now)
- **UC5** A release breaks something and we yank it: `setup --update --pin=v2026.05.31` everywhere; `gh release delete v2026.06.01` with rationale; on-call writes postmortem.
- **UC6** A developer revokes consent: one config flag stops uploads; one CLI flag (`ai-sherpa wipe-me`) triggers server-side `DELETE FROM traces WHERE developer_id = ?`.
- **UC7** A developer requests their data: `ai-sherpa what-do-you-have-on-me` returns a JSON snapshot of every trace, finding, and Issue tied to them.
- **UC8** Server is down for 3 days: laptop OTel Collectors buffer via `file_storage` extension; resume on reconnect; documented RPO/RTO in the runbook.

## 9. What we explicitly are NOT going to do

This list is as important as the deliverables list. Each item is something the previous program would have built and should not.

- **Build a FastAPI ingest service.** OTel Collector is the ingest. Langfuse is the application.
- **Build a SQLite FTS5 index or a ChromaDB embedding tier.** ClickHouse via Langfuse covers both.
- **Build a Google Apps Script mailer with a shared secret.** Atom feed + Google Group's built-in feed-to-email.
- **Build `generate-release-notes.sh` + pandoc + jq pipes.** GitHub-native `.github/release.yml` with `categories:`.
- **Build a centroid-based recurrence detector.** Phase 2 uses a deterministic fingerprint (`scenario_id + domain + sorted top-N contributing developer hashes`); embedding-based similarity is a v2 question, not a Phase 2 commitment.
- **Build a 14-detector taxonomy as a monolith.** Phase 2 ships 4–6 high-confidence Scorers drawn from Phase 1 evidence; rest are deferred until thresholds can be calibrated.
- **Pre-commit to "10+ years storage."** We commit 1 year, with re-review.
- **Pre-commit to a "Phase 2b" of LLM polish.** Langfuse evals cover this if needed.
- **Treat "intranet only" as a security control.** mTLS + bearer tokens are not optional in Gate E.

## 10.5 Developer-facing affordances (the honest caveat)

The architect review observed that v1 was "inward-facing infrastructure dressed as a DX initiative." v2 reduces that critique but does not eliminate it. The Scorer Registry is the right *architectural* contribution — it fills a real OSS gap and gives the central team measurable leverage — but it is not by itself a *developer-experience* contribution. The 150 individual developers should get something concrete back beyond "a better version next week."

Four affordances make the DX layer real. The first two are committed in Phase 1; the last two are named here so they're on the table at Gate D, not invented late.

### 10.5.1 `/ai-sherpa-feedback` *(Phase 1, committed)*
Already in §5.1. One keystroke from inside a session to a structured Issue. **Direct value:** "I noticed something wrong; reporting it took 45 seconds, not 15 minutes."

### 10.5.2 `tools/local-session-analyzer` *(Phase 1, committed, free byproduct of §5.9)*
Run locally against your own `~/.claude/projects/`. No fleet telemetry. Outputs personal Read:Edit ratio, self-admitted error count, top rules violated this week, etc. **Direct value:** "I can see how I'm using Claude without opting into anything."

### 10.5.3 `/ai-sherpa-insights` *(Gate E or later, sketched not committed)*
Slash command that surfaces *the developer's own* Langfuse data, assuming they opted in to telemetry. "Show me my last week", "compare me to my domain baseline (anonymized)", "which CLAUDE.md rule am I tripping on most." **Direct value:** "I get analytics about my own work, not just contribute to a fleet metric."

### 10.5.4 Weekly personal digest *(Gate E or later, sketched not committed)*
Opt-in email per dev with their own metrics + the team improvements that landed this week + "rules you helped surface" credit. Inverts the framing from surveillance to participation. **Direct value:** "I see what my data is being used for, and I get credit when it matters."

### What this caveat means for the program

v2 is correct about what the central team should build. It is not by itself a complete answer to "what does each dev get back?" **Gate D should commit at least one of §10.5.3 or §10.5.4** — not both, but not zero — before Gate E ships. The decision data is whatever opt-in rate Phase 1 produces: if it's < 50%, the DX affordances become a Phase 2 prerequisite, not a stretch goal.

The honest version of v2's pitch to a developer:

> *"Phase 1 gives you a one-keystroke feedback button and a local analyzer. Phase 2 will give you personal insights from your own data once we're confident the system works. We're not asking you to opt into pure surveillance — we're asking you to opt into a loop you also benefit from."*

That sentence — verbatim — belongs in `docs/feedback-guide.md` and in the consent prompt the collector shows on first run. If we can't say it honestly, the consent posture is wrong and Gate E should not start.
